OpenGL and C++ Graphics Simulation

SYSTEM ARCHITECTURE DEEP-DIVE

Technical Implementation

Throughout this module I was tasked with 4 worksheets to complete, each building off of the last one:

• Worksheets 1 & 2 (Prototyping): Rendered 3D models into the scene, implemented camera controls, and researched a Lorenz Attractor using particles. Created a simple 2D particle simulation. Profiling inside Visual Studio showed stable performance at 160 FPS up to ~125,000 particles, dropping to 15 FPS at 100,000 pixels due to performance bottlenecks.

• Worksheet 3 (3D Transition & Profiling): Shifted the simulation into a 3D environment using chaos equations. Profiling revealed heavy CPU-to-GPU rendering inefficiencies, with uneven CPU thread utilization causing thread stalling. Conducted memory profiling to prevent memory leaks and ensure correct C++ garbage collection.

• Worksheet 4 (Optimization & Interaction): Resolved GPU bottlenecks by implementing multithreading and Vertex Buffer Objects (VBOs), sending data in unified buffers directly to GPU memory. Locked simulation at 60 FPS for consistency. Post-optimization testing achieved 40 FPS at 1M particles, 20 FPS at 2M, and 10 FPS at 3M. Implemented ImGui visual interface sliders for live user interaction with the chaos equation constants.

The Challenge

During this project I struggled during multiple stages. At first I had issues deciding what to make; my original idea was to make a hair simulation using particle chains, but a lack of accessible documentation forced a strategic pivot. After further research, I shifted focus entirely to particle-based mathematical attractors.

I also faced significant time management challenges, balancing this research alongside my dissertation and a concurrent group project.

Project Management & Methodology

• Personal Agility & Timeboxing: To manage three intensive academic modules simultaneously, I utilized Agile methodologies to build a personal Kanban task board, establishing a rotating daily schedule to ensure disciplined resource allocation and hit all deadlines.

• Scope Management: During the optimization phase, I identified further ways to reduce GPU calls and precalculate equation loops. However, applying Google PM principles, I classified these features as "out of scope" for the strict project time limit to guarantee the delivery of a stable, polished MVP.

BSc Dissertation - To What Extent are Users Able to Recognise Personality Traits from a Chatbot?

SYSTEM ARCHITECTURE DEEP-DIVE

Technical Implementation

• Local Architecture: Deployed an entirely local, converstational AI framework using Ollama to host a DeepSeek 8-Billion parameter model, ensuring zero API dependency and local data control.

• Software & Pipeline Integration: Utilised LangChain to create a Python pipeline to pass complex promp architectures dynamically into the model, alongside a user interface creating using Tkinter.

• Regex Output Parsing: Created a real-time output cleaning middleware function, utilising Python's built-in regex library to capture, clean, and strip raw LLM think tokens out of the interface stream to provide clean, believable user dialogue.

• Test-Driven Engineering: Developed and ran a test suite using the Pytest framework, inheriting standard Unit Test and Test Case structures to build automated testing scripts checking for bot invocation response, empty string bounds, character inputs, and edge cases.

The Challenge

• Hardware & Token Latency Bottlenecks: Encounted substantial prompt processing delays and non-existent software bug loops during development due to immense hardware resource overhead of hosting a local reasoning models on consumer graphics configurations.

• Technical Resolution: Performance degradation was mitigated by benchmarking token delivery speeds and response times to dynamically pivot to a tightly scoped 8B model size and disable active contextual chat memory buffers to prevent token context build up. This drastically decreased respinse generation latency.

• Data Storage & Privacy Ethics Constraints: Faced strict data collection guidelines during participant interactions. I adhered to the UK's GDPR, Data Protection Act, the Nuremberg Code, and the BCS Code of Conduct.

• Data Protection Solution: Overcame the privacy risks by turning off LLM cahcing features to prevent physical storage of personal inputs, designing a anonymouse data pipeline using Microsoft Forms, and scrubbing identifiable metrics from final evaluation files.

Project Management & Production

• Independent Lifecycle Ownership: Structured, scheduled, and delivered a multifaceted, multi-month research roadmap follwing strict academic milestones across initial planning, AI development, User testing, Data sourcing, and reporting.

• Agile Production & QA Pipelines: Practiced an iterative, incremental development workflow, utilising the Booch Method, to ensure consistent code progression. A/B prompt testing was used to map baseline prompt variations against optimal personality parameters.

• Advanced Risk & Release Management: Deployed a securem remote version-control infrastructure via GitHub to enfore structed refactoring, logging bugfixesm feature tasks, and technical risk updates natively through commit logs.

• Quantitative Methodology & Analytics: Directed participant logistics to coordinate a 60 question Likert-scale HEXACO evaluation questionaire. Exported results to secure CSV sheets and mapped raw scores via RStudio to visulise data and carry out One-Sample T-Tests to calculate confidence intervals and statistical significance.
